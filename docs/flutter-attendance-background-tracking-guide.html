Flutter Attendance Background Tracking Guide

This document explains how to build the employee mobile app for the new attendance flow: login, home/profile, check-in, check-out, attendance history, and continuous background location tracking from check-in until check-out.

Login → Bootstrap → Check policy/device/IP/location → Check In → Start foreground/background tracking → Auto sync pending items → Check Out → Stop tracking → History

2. What The App Should Do

Pre-Punch Validation

Load employee policy, branch, shift, allowed IP patterns, geofence rules, selfie rules, and device-binding rules before allowing a punch.

Active Session Tracking

After check-in, keep tracking live location until check-out. Send online pings immediately or queue them offline.

Offline Recovery

Store punch events and location pings in local SQLite when offline, then sync them in order when internet returns.

3. High-Level Mobile Flow

1. Employee logs in 2. Call /attendance/mobile/bootstrap 3. Store: - employee profile - shift - branch geofence - allowed branch IP patterns - attendance policy - current attendance status 4. Call /attendance/mobile/request-ip 5. Detect: - current GPS - current device ID - current request IP - selfie readiness 6. Before punch: - verify GPS if policy requires it - verify branch geofence if policy requires it - verify request IP if office attendance requires allowed Wi-Fi/IP - verify selfie if policy requires it - verify device binding if policy requires it 7. Call /attendance/mobile/punch 8. If check-in succeeds and session stays active: - start background tracking loop - start online/offline sync worker 9. While checked in: - call /attendance/mobile/location-ping on interval or movement threshold - refresh /attendance/mobile/active-session periodically 10. On check-out: - call /attendance/mobile/punch with check_out - stop background tracking - flush pending queue if online

4. Main APIs

POST/auth/login

Login using email or phone and password.

{ "login": "employee@gmail.com", "password": "employee@123", "remember": true }

GET/attendance/mobile/bootstrap

Load employee dashboard essentials immediately after login.

Important response fields to store:

data.company.timezone data.employee.employee_code data.employee.branch_name data.employee.branch_latitude data.employee.branch_longitude data.employee.geofence_radius_meters data.employee.shift_name data.employee.start_time data.employee.end_time data.employee.gps_required data.employee.selfie_required data.employee.checkout_selfie_required data.employee.device_binding_required data.employee.geofence_required data.employee.wifi_ip_restriction_required data.employee.continuous_tracking_enabled data.employee.continuous_tracking_interval_seconds data.employee.allow_field_attendance data.employee.allow_wfh_attendance data.branch_networks data.today_attendance

GET/attendance/mobile/request-ip

Get the server-seen request IP. Use this before every punch and during active session refresh.

{ "status": "success", "data": { "request_ip": "103.120.12.10", "is_loopback": false, "hint": null, "server_time": "2026-05-09T12:00:00+05:30" } }

GET/attendance/mobile/active-session

Use this to know whether the employee is currently checked in and to load recent tracks already stored by the server.

POST/attendance/mobile/punch

Used for both check-in and check-out.

Recommended multipart payload:

punch_type=check_in | check_out attendance_mode=online | offline work_mode=office | field | wfh | hybrid occurred_at=2026-05-09T09:58:02+05:30 latitude=22.5726451 longitude=88.3638891 gps_accuracy_meters=12 location_text=22.57265, 88.36389 (±12m) device_id=DEVICE_UNIQUE_ID device_name=Samsung Galaxy M14 device_platform=android device_model=SM-M145F os_version=Android 14 app_version=1.0.0 battery_level=83 network_type=wifi internet_status=online remarks=Optional note selfie=(multipart file)
If the API returns exceptions like outside_geofence, wifi_ip_not_allowed, device_not_bound, or selfie_missing, the app should show the reason and stop silent success handling.

POST/attendance/mobile/location-ping

This is the main tracking API for silent location updates after check-in.

{ "attendance_id": 23, "recorded_at": "2026-05-09T10:21:14+05:30", "latitude": 22.5727012, "longitude": 88.3639011, "gps_accuracy_meters": 18, "battery_level": 79, "network_type": "wifi", "is_background": true, "speed_kmph": 0.0, "source": "flutter_background_service", "sync_status": "synced", "local_queue_id": "trk_1746775874_001", "metadata": { "provider": "gps", "app_state": "background" } }

Response:

{ "status": "success", "message": "Location ping recorded.", "data": { "id": 154, "employee_attendance_id": 23, "recorded_at": "2026-05-09 10:21:14", "latitude": "22.5727012", "longitude": "88.3639011", "network_type": "wifi", "source": "flutter_background_service", "sync_status": "synced" } }

POST/attendance/mobile/activity-log/sync

Use this to batch meaningful employee activity events instead of calling the server for every tiny UI action.

{ "items": [ { "activity": "dashboard_opened", "title": "Dashboard opened", "description": "Employee opened the attendance dashboard", "attendance_id": 23, "occurred_at": "2026-05-09T10:01:12+05:30", "source": "flutter_employee_app", "local_queue_id": "act_1746774672_001", "session_key": "sess_1746774672_ab12", "category": "attendance_session", "severity": "info", "data": { "network_type": "wifi", "request_ip": "103.120.12.10", "location": "22.57265, 88.36389 (±12m)" } }, { "activity": "gps_unavailable", "title": "GPS unavailable", "attendance_id": 23, "occurred_at": "2026-05-09T10:09:42+05:30", "source": "flutter_employee_app", "local_queue_id": "act_1746775182_002", "category": "attendance_session", "severity": "warn", "data": { "reason": "location_permission_denied" } } ] }

Recommended events:

  • dashboard_opened
  • dashboard_resumed
  • network_online
  • network_offline
  • network_changed
  • gps_acquired
  • gps_unavailable
  • tracking_started
  • movement_tracked
  • tracking_stopped
  • punch_attempted
  • punch_captured
  • attendance_flagged_for_review

POST/attendance/mobile/sync

Use this when offline punch events or offline tracking rows need to be synced later.

Recommended use:

  • Queue offline punch items in SQLite
  • Queue offline tracking pings in SQLite
  • When internet returns, send them in chronological order

GET/attendance/mobile/history

Use this for the employee attendance history screen.

5. Background Tracking Rules

CaseRecommended App Behavior
Employee not checked inDo not run silent tracking service.
Employee checked inStart foreground/background tracking worker.
Policy continuous tracking disabledDo not start background worker. Only send punch-time location.
Policy interval availableUse continuous_tracking_interval_seconds from bootstrap.
App onlineSend pings directly to /attendance/mobile/location-ping.
App offlineQueue pings in SQLite with local queue IDs.
Internet restoredSync queued items oldest first.
Employee checked outStop background worker and clear active tracking state.

6. Recommended Tracking Strategy

Foreground

  • Refresh request IP every 45 seconds.
  • Refresh active session every 45 seconds.
  • Send location ping on configured interval.
  • Also send a ping if movement threshold is crossed, for example 20 meters.

Background

  • Run a background service only after check-in.
  • Use device location callbacks to capture movement.
  • Save queue items locally if network fails.
  • Mark background pings with is_background=true.

9. Important UI States

Before Check-In

Show employee profile, branch, shift, policy badges, current GPS, current IP, device readiness, and selfie readiness.

During Active Session

Show checked-in time, live current location, last ping time, network/IP state, tracking state, and queue state.

After Check-Out

Show final check-out time, total worked duration, last location, sync state, and that tracking has stopped.

10. Admin/HR Detail View The Mobile App Supports

The backend now stores enough detail for web admin/HR to inspect:

  • current employee location while session is active
  • movement trail from check-in to check-out
  • check-in selfie and check-out selfie
  • request IP used during punch
  • network type, sync status, and exception reasons
  • approval history for suspicious records

That means the mobile app should keep the tracking payload rich and consistent, especially attendance_id, recorded_at, latitude, longitude, network_type, battery_level, is_background, and local_queue_id.

11. Final Recommended Implementation Order

Phase 1 - Login - Bootstrap - Employee home/profile - Check-in - Check-out - Attendance history Phase 2 - Active session refresh - Request IP refresh - Device ID binding - Camera selfie capture Phase 3 - Background location tracking - SQLite offline queue - Auto sync worker - App resume/connectivity recovery Phase 4 - Better fraud prevention - Better battery optimization handling - More detailed local tracking analytics
